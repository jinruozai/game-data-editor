# 数据编辑器需求文档

本文档描述通用 **GameData 编辑器** 的需求：数据格式、内部数据模型、读取/校验/归一化/保存规则与日志要求。

---

## 1. 数据格式（JSON）

### 1.1 目录与文件
- 编辑器是通用工具：可以打开任意目录。
- 编辑器在加载目录时，会在目标目录根查找可选的 `*.gmdata` 配置文件；存在则读取，不存在不影响加载。
- 编辑器会递归扫描目录（及子目录）下的 `.json` 文件，并按本节“表格 JSON 格式”识别可加载的文件。

### 1.2 单表格式（统一）
每个 JSON 文件都是一个“ID → 对象”的字典，并且顶层必须包含版本化的结构定义 `struct_def_<版本号>`：

```jsonc
{
  "struct_def_0": { /* 字段类型定义（TypeConfigItem） */ },
  "1234567890123": { /* 实体数据 */ },
  "2345678901234": { /* 实体数据 */ }
}
```

> 说明：编辑器当前只读取 `struct_def_<version>`，其中 `version` 由 `*.gmdata.version` 指定；缺省 `version = 0`。

#### struct_def_<n>（字段 → 字段定义 JSON）
`struct_def_<n>` 是一个字典：key 是字段名，value 是“字段定义 JSON”：

- `type`：**必填**，表示使用哪个 TypeConfig（`TypeConfig[type]`）作为该字段的类型定义来源
- 允许写任意“覆盖字段”（可选）：用于覆盖 TypeConfig 中的默认配置（例如覆盖 `default`、`type_agv.options`、`type_render` 等）
- `default`：**可选**。不写时使用 `TypeConfig[type].default`；需要覆盖默认值时才写 `default`

字段定义支持覆盖的字段集合以 `TypeConfigItem` 为准（参见 `doc/game-data-editor/doc/数据类型.md`）：
`base_type / type_render / default / mem / type_agv / support_render / struct_def`。

> 另外：TypeConfigItem/字段定义还可以包含 `card_style`，用于“卡片模式”展示布局（见《数据类型.md》5.card_style）。

示例（常用字段示意，省略部分可选字段）：
```jsonc
"struct_def_0": {
  "name": { "type": "string", "mem": "名称" },
  "icon": { "type": "img", "mem": "图标（binary 或 project://）" },
  "description": { "type": "string", "type_render": "textarea", "mem": "描述" },

  "quality": {
    "type": "enum_int",
    "mem": "品质",
    "type_agv": { "options": { "0": "凡品", "1": "稀有", "2": "典藏", "3": "限定", "4": "隐藏" } }
  },

  "tags": { "type": "array", "type_render": "tags", "mem": "标签ID数组（int64）" },

  "property": {
    "type": "array",
    "type_render": "list",
    "mem": "属性列表（[[id,num],...]）"
  }
}
```

> 默认值说明：
> - 字段若只写 `{ "type": "xxx" }`，则该字段默认值来自 `TypeConfig[xxx].default`
> - 若要覆盖默认值，可在字段定义里写 `default`（会覆盖 TypeConfig 中的 default）

### 1.3 ID（新增数据时的生成规则）

编辑器新增实体时，需要生成一个全局唯一的 int64 ID（显示/写入时为“数字字符串”）。要求：
- 全局唯一、非 0
- 不体现顺序关系（禁止递增序列）

推荐生成算法（简单、无顺序感）：
- 生成一个随机的 int64 正整数（范围：`1..2^63-1`）
- 与当前已加载的 `gameData` 做冲突检测；若冲突则重试
- 建议最多重试 N 次（例如 100 次），仍冲突则报错并提示用户

> 备注：实现上可以用安全随机数（如 `crypto.getRandomValues` 或系统随机源），保证分布均匀、不可预测。

> 说明：编辑器作为通用工具，除 ID 生成与重复检测外，不内置任何固定业务字段规则；字段类型/校验由 `struct_def` 与 `type_config` 决定。

## 2. 编辑器功能需求

### 2.1 打开与加载
编辑器支持两种打开方式（等价）：
- 打开目录：选择任意目录 `root`
- 打开配置文件：选择任意 `*.gmdata` 文件（等价于打开其所在目录）

加载步骤：
1) 读取目录根下的 `*.gmdata`（若存在；用于 `version` 与 `type_config`）
2) 递归扫描 `root` 下所有 `.json`，仅加载包含 `struct_def_<version>` 的文件为“表”
3) 将所有表的数据汇总到内存模型（见 2.2），并在 log 记录加载统计与异常

### 2.2 编辑器内部数据模型（内存）
编辑器运行时只需要维护以下核心数据（都在内存中）：

#### 2.2.1 version（number）
- 含义：当前使用的 `struct_def_<version>` 版本号
- 来源：`.gmdata.version`，没有则 `version = 0`

#### 2.2.2 内置 type_config（object）
- 结构：`builtin_type_config: { [type: string]: TypeConfigItem }`
- 含义：编辑器自带的默认类型定义（不可编辑）
- TypeConfigItem 的字段见《数据类型.md》

#### 2.2.3 项目 type_config（object）
- 结构：`project_type_config: { [type: string]: TypeConfigItem }`
- 含义：用户在当前目录下通过 `.gmdata.type_config` 定义/覆盖的类型

#### 2.2.4 gameData（全局实体字典）
- 结构：`gameData: { [id: string]: any }`
- 含义：把所有表里出现的实体都拉平成一个全局字典（ID → 数据对象）

#### 2.2.5 tableMap（表索引 + 表结构）
`tableMap` 以“相对路径名（无 .json 后缀）”为 key，把每张表的结构与 ID 列表收拢到一起：

```jsonc
tableMap = {
  "相对路径名": {
    "struct_def": { /* 来自文件中的 struct_def_<version>（内存里不带版本后缀） */ },
    "id": ["3233", "433434"]
  }
}
```

- `struct_def`: `{ [fieldName: string]: { type: string, ...覆盖项 } }`（其中 `type` 必填；覆盖项字段与 TypeConfigItem 同名即可）
- `id`: 该表包含的 ID 列表（顺序即表内顺序）

#### 2.2.6 字段最终类型定义（覆盖优先级）
对某个表的某个字段：
`最终字段类型定义 = struct_def 覆盖 ( 项目 type_config 覆盖 ( 内置 type_config[字段.type] ) )`

### 2.3 struct_def 版本规则
- `version` 来自 `*.gmdata.version`，缺省为 `0`
- 加载某个 JSON 表时必须存在 `struct_def_<version>`；否则该文件不作为表加载，并在 log 提示原因

### 2.4 渲染与编辑规则（type_render）
字段渲染遵循三态回退：
1) schema 模式：
   - 读取 `struct_def_<version>[field]`（FieldDef，且 `type` 必填）
   - 从合并后的 `typeConfig[type]` 取类型定义（默认值/渲染器/参数）
   - 将 FieldDef 中的覆盖项（如 `default/type_render/type_agv/...`）覆盖到该 TypeConfig 上，得到该字段最终渲染配置
2) 基础回退：若 `type` 在 typeConfig 中找不到（未知 type），回退为 `input`（并标记错误）
3) 兜底：渲染器运行异常/参数不合法 → json 原始编辑器

### 2.5 加载时校验与日志（log）
加载 JSON 时必须做两类检查，并在 log 中提示：

1) **数据符合 struct_def**
   - 对每张表：表内每条实体数据都要符合该表 `struct_def`（字段类型由 `type` 对应的 TypeConfig 决定）
   - 校验规则来自 `typeConfig[type]`（例如 enum/range/img/snd/bool 等）
   - 不符合则在 log 提示：表名/ID/字段/原因（不中断加载）

2) **全局 ID 不重复**
   - 加载所有表时，如果发现同一个 ID 在不同表（或同表）出现多次，log 提示重复 ID（不中断加载，但需标红/定位）

### 2.6 值归一化（加载时强制转换）
编辑器**不做字段名检测**，只对 value 做归一化：当 value 不符合 `type_config` 定义的基础类型时，尝试强制转换。

约束：底层基础类型只有四种：`int` / `float` / `string` / `array`。

归一化规则（示意）：
- 目标 `int`：尽量把 value 转成整数；失败则置为 `0`
- 目标 `float`：尽量把 value 转成浮点；失败则置为 `0.0`
- 目标 `string`：尽量把 value 转成字符串；失败则置为 `""`
- 目标 `array`：若为数组则保留；若为单值则转为单元素数组；失败则置为 `[]`

日志（log）要求：
- 发生了转换：log 记录“表名/ID/字段/原值/新值/目标类型”
- 转换失败并回退为空值：log 记录“表名/ID/字段/原值/目标类型/使用空值”

### 2.7 保存（核心流程）

保存分两步：先写 `*.gmdata`，再写各个表的 JSON。

**A. 构造并写入 `.gmdata`（可选）**
1) 构造 `outGmdata = {}`（空对象）
2) 若 `version != 0`：写入 `outGmdata.version = version`
3) 若 `projectTypeConfig` 非空：
   - 对其中每个 type，与 `builtinTypeConfig[type]` 做字段级比较
   - 仅把“与内置不同”的 type 记录到 `outGmdata.type_config`
4) 若 `outGmdata` 非空：写入/覆盖 `.gmdata` 文件；否则不写（或可选择删除旧 `.gmdata`）

**B. 写入各个表文件**
对每个 `pathKey -> idList`：
1) 构造表 JSON：
   - 写入 `struct_def_<version>`（来自 `tableMap[pathKey].struct_def`，写回时再加版本后缀）
   - 对 `tableMap[pathKey].id` 中每个 id：写入 `json[id] = gameData[id]`
2) 写入到 `pathKey + ".json"`（相对 root 的路径）

示例：
- `pathKey = "盲盒/华廷录"`
- `tableMap["盲盒/华廷录"] = [3233, 433434, ...]`
- 保存时写入文件：`盲盒/华廷录.json`，内容包含 `3233、433434` 等 ID 对应的数据。

## 3. `*.gmdata` 数据格式

`*.gmdata` 指的是文件后缀名（不是固定文件名）。目录中任意 `*.gmdata` 文件都可作为该目录的配置文件，用于保存用户自定义 TypeConfig。每一项都可选。

```jsonc
{
  "version": 0,
  "type_config": {
    "enum_quality": {
      "base_type": "int",
      "type_render": "enum",
      "default": 0,
      "mem": "自定义品质枚举",
      "type_agv": { "options": { "0": "凡品", "1": "稀有" } }
    }
  },
  "editor_cache": {
    "layout": {
      "preset": "default",
      "dock": {}
    },
    "table_view": {
      "mode": "card",
      "sort": { "field": "name", "order": "asc" }
    }
  }
}
```

- `version`: struct_def 版本号（number）。编辑器加载 JSON 时会寻找 `struct_def_<version>`；未提供 `.gmdata` 时默认 `version = 0`。
- `type_config`: 用户自定义 TypeConfig 字典（key 为 type 名，value 为 TypeConfigItem）
- `editor_cache`: 编辑器缓存数据（可选）。用于保存布局、面板状态、表格显示模式/排序等“零碎但影响体验”的信息；不参与运行时数据结构，仅供编辑器恢复现场。

> 运行时合并与覆盖规则：`struct_def 覆盖 ( gmdata 覆盖 ( 内置 TypeConfig[type] ) )`（详见 2.4 组件三态中的“schema 模式”）

## 4. 导出 TypeConfig（可选功能）
编辑器可提供“导出当前项目可用的 TypeConfig”功能：
- 导出内容：`merge(builtin_type_config, project_type_config)`
- 导出为单独 JSON 文件（用于分发/复用），不要求写入 `*.gmdata`
